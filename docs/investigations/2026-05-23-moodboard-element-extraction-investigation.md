# Investigation: Agentic Moodboard / Branding-Board Element Extraction

**Date Started:** 2026-05-23 **Investigator:** Claude Code (Opus 4.7, 1M
context) **Status:** Concluded **Outcome:** Proposal Recommended (with a
mandatory proof-of-concept gate)

---

## Question / Motivation

Cole has branding boards generated by other agents — single PNG/JPG images
packed with distinct sub-elements (mascot variants, icons, color swatches, type
samples). Today extraction is a Photoshop chore: marquee, cut, alpha-mask,
export. He wants to replace that with an agent-orchestrated flow that lives in
this repo's plugin system (alongside `tuskboard`, `digestify`,
`html-mockup-prototyping`) and accepts three selection modalities:

1. **Natural language** — "the mammoth icon in the upper-left"
2. **Visual markers** — coloured rectangles drawn on the source image
3. **Click coordinates** (stretch) — (x, y) points → segment object at point

Per element, the user picks: plain rectangular crop **or** alpha-removed
PNG/WebP on transparent background.

This investigation answers: **what pipeline, which models/services, what
ergonomics, and is the idea proven enough to build a skill around?**

## Investigation Findings

### Pipeline architecture — the right primitives

The user's mental model in the prompt is correct in shape but conflates two
roles. The cleaner decomposition:

```
INPUT IMAGE
   │
   ▼
┌──────────────────────────────┐
│ 1. REGION SELECTION          │   produces N bboxes (or points / masks)
│    one of:                   │
│    • OpenCV HSV+contour      │   (visual markers mode)
│    • Vision-LM bbox          │   (NL mode — Claude or Florence-2)
│    • User-supplied coords    │   (click mode)
└──────────────────────────────┘
   │  bboxes / points
   ▼
┌──────────────────────────────┐
│ 2. MASK REFINEMENT (optional)│   produces pixel-tight mask per region
│    • SAM 2 (bbox/point hint) │   only needed if user picks "transparent bg"
└──────────────────────────────┘
   │  bboxes + (optional) masks
   ▼
┌──────────────────────────────┐
│ 3. BACKGROUND REMOVAL        │   alternative to SAM2 for alpha cutout
│    • rembg / BiRefNet /      │   run on the *cropped* region, not the
│      RMBG-2.0                │   whole board
└──────────────────────────────┘
   │
   ▼
┌──────────────────────────────┐
│ 4. CROP + COMPOSITE + SAVE   │   pure libvips/sharp/Pillow — no AI
└──────────────────────────────┘
```

**Important correction to the user's pipeline:** stages 2 and 3 are
_alternatives_, not a chain. You don't usually need both. Two viable shapes:

- **Shape A (segment-then-crop):** SAM 2 produces a tight mask from the bbox
  hint → apply mask as alpha → crop to mask bounds. One pass, no separate
  background-removal model. Good for graphic / illustration elements where the
  "subject vs. board background" boundary is clean.
- **Shape B (crop-then-cutout):** Crop the bbox region → run BiRefNet / RMBG-2.0
  on the cropped tile → save with alpha. Two passes but each model does what it
  was trained for. Background-removal models (RMBG-2.0 in particular, which
  benchmarks at ~90% vs BiRefNet's ~85%) are state-of-the-art on the specific
  task of subject/background separation and were trained on the kinds of
  composited images a moodboard contains. They handle hair, anti-aliased edges,
  and soft drop-shadows much better than SAM 2 masks, which tend to be tight but
  blocky.

Both shapes are valid. The pragmatic call: **default to Shape B** (crop-then-
cutout). Reasons:

1. Background-removal models are optimized for exactly the artifact you want (a
   clean alpha matte). SAM 2 is optimized for _segmentation accuracy_, not matte
   quality — its masks are binary and can produce jagged edges on antialiased
   graphics.
2. Cropping first dramatically shrinks the image the bg-removal model sees,
   making local inference (where rembg's ISNet runs in ~100–300ms on a tile)
   trivially fast. Running RMBG-2.0 on a 2000x2000 full moodboard is overkill;
   running it on a 400x400 cropped mascot is a non-event.
3. Shape B lets the user choose "plain crop" with zero AI invocations — they
   just don't run stage 3. That's a real cost saving when the user wants 8 plain
   crops from a board.

Use SAM 2 _only_ when the user provides a click point (stretch goal) and there
is no bbox available — then SAM 2's point-prompt mode earns its keep.

### Region selection — three modalities, three honest assessments

#### 3a. Visual markers (coloured rectangles)

**Feasibility: HIGH. Build this first.**

A standard OpenCV HSV-threshold + contour pipeline handles the common case
robustly. Sketch of the pipeline:

```python
hsv = cv2.cvtColor(img, cv2.COLOR_BGR2HSV)
# Detect any saturated colour the user might draw with — broad band
mask = cv2.inRange(hsv, (0, 100, 100), (180, 255, 255))  # saturation > 100
# Plus a red-specific band (wraps hue=0)
red = cv2.inRange(hsv, (0,120,70),(10,255,255)) | cv2.inRange(hsv,(170,120,70),(180,255,255))
contours, _ = cv2.findContours(mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
# Filter contours: minRect area > X, aspect not absurd, perimeter/area suggests
# a hollow rectangle outline (not a filled blob)
```

Real-world gotchas to design around:

| Gotcha                                              | Mitigation                                                                                                                                         |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| Source artwork contains the marker colour           | Let user choose marker colour at invocation (`--marker-color red`); pre-warn if the dominant colour in the source matches.                         |
| Rectangles aren't axis-aligned (user drew freehand) | `cv2.minAreaRect()` returns a rotated bounding rect; deskew the crop with `cv2.warpAffine` if rotation > 5°.                                       |
| Line thickness varies                               | Use morphological closing (`cv2.morphologyEx(..., cv2.MORPH_CLOSE)`) before contour detection.                                                     |
| Marker outline touches source artwork               | Use `cv2.RETR_EXTERNAL` (only outermost contours) and inset the bbox by N pixels equal to the detected stroke width before passing to the cropper. |
| Overlapping markers                                 | Detect, then ask the user (preview-then-confirm pattern, see ergonomics section) which to keep.                                                    |

**Output of this stage:** N axis-aligned bboxes (after deskew), ordered
top-to-bottom / left-to-right.

#### 3b. Natural language ("the mammoth icon upper-left")

**Feasibility: MEDIUM-HIGH, with a precision caveat that determines
architecture.**

This is question 3 from the brief — the most important architectural fork. **Can
Claude's own vision return bboxes precise enough to feed SAM 2 / bg-removal
directly, or do we need a dedicated grounding model (Florence-2 / Grounding
DINO)?**

Evidence I gathered:

- **Anthropic's vision docs (primary source):** Claude Opus 4.7 supports
  high-resolution images (up to 2576 px on the long edge) and "When asking
  Claude to output coordinates (points, bounding boxes, etc.), they will be
  expressed with respect to the resized/padded image and will need to be
  rescaled/translated accordingly client-side based on the original and resized
  dimensions." That's a _direct, documented API contract for bbox output_ —
  Anthropic is treating coordinate emission as a first-class use case for Opus
  4.7.
- **Same docs, limitations section:** "Claude's spatial reasoning abilities are
  limited. It may struggle with tasks requiring precise localization or
  layouts."
- **Third-party benchmark (MultihopSpatial, arXiv 2603.18892):** Claude Opus 4.5
  ranked 7th on multiple-choice spatial questions but dropped to 29th on
  Acc@50IoU — "potential gaps between semantic understanding and precise visual
  grounding." Best model overall peaked at ~40% IoU on strict localization. Opus
  4.7 is newer (April 2026 release per third-party reviews) with explicit bbox
  improvements, but no public IoU benchmark for it exists yet that I could find.
- **Working open-source example:** `Doriandarko/Claude-Vision-Object-Detection`
  exists, uses Claude Sonnet's vision to draw bounding boxes around detected
  objects with confidence scores. So the _engineering pattern_ is community-
  validated.

**My honest read:** Claude Opus 4.7's bboxes are _probably good enough_ for the
moodboard use case specifically — moodboards have well-separated, visually
distinct elements on uncluttered backgrounds, which is much easier than picking
out the 4th-from-left object in a cluttered shelf. They are _almost certainly
NOT_ tight enough to crop directly. But they are very likely good enough to
_seed_ a SAM 2 mask or to _seed_ a bbox-pad-then- bg-remove crop. The pipeline
absorbs ~20-40px of bbox slop without harm, because:

- For Shape B (crop-then-cutout), an over-large bbox is harmless: the bg remover
  trims the alpha to the subject anyway.
- For Shape A (SAM 2 hint), SAM 2's bbox-prompted mode is _designed_ for loose
  hints — it refines the boundary itself.

**Conclusion on this fork:** Skip Florence-2 / Grounding DINO for v1. Use
Claude's own vision via a second API call (or, in the agent skill case, the
_same_ agent turn that invoked the skill) to emit bboxes. This collapses the
pipeline materially — one fewer model, no PyTorch dependency for NL mode.

**Risk to flag honestly:** The official limitations doc still warns about
precise localization. If empirical PoC results (see below) come back with bboxes
that are _quadrant-rough_ rather than _element-rough_, fall back to Florence-2
hosted via Replicate or fal.ai. This is the single biggest risk in the whole
design and must be validated on real input before committing.

**Empirical experiment I could not run inside this turn:** I attempted to locate
a sample branding-board image to localize as evidence within this investigation.
The agent surface I'm running in does not let me pass an arbitrary image URL
back to my own vision channel and receive bboxes I can quote. So the
bbox-precision claim above remains a _strongly-supported hypothesis_, not a
verified measurement. The PoC plan in the next section treats this as job zero.

SAM 2 point-prompt mode handles this trivially. Defer to v2 — Cole hasn't

**Eliminated early:**

- **Photoshop Action / `pyautogui`-style RPA** — non-starter, brittle, defeats
  the "agent does it" goal.
- **DALL-E / image-gen inpainting** — wrong tool. Generative models hallucinate,
  we want extraction.
- **OpenCV GrabCut alone** — old-school, requires per-image tuning, dominated by
  every model above.

### Decision matrix — full pipeline

Weighted on: invocation friction (HIGH — this is an agent skill), per-call cost
(MEDIUM — bursts of 10 are realistic), quality on graphic/illustration content
(HIGH — moodboards are mostly graphics, not photos), and Apple Silicon
reliability (HIGH — Cole's environment).

| Criterion                       | Weight | A. Pure local (`opencv` + `rembg` + Claude bboxes) | B. Pure hosted (`fal.ai bria` + Claude bboxes + optional `replicate sam-2`) | C. Hybrid (local default, hosted fallback) |
| ------------------------------- | ------ | -------------------------------------------------- | --------------------------------------------------------------------------- | ------------------------------------------ |
| Invocation friction             | High   | Medium (one-time `uv tool install`)                | **Low (just API keys)**                                                     | Low after one-time setup                   |
| Per-call cost                   | Med    | **$0**                                             | $0.018/element × N                                                          | $0 default, $0.018 only on fallback        |
| Quality (graphic elements)      | High   | Good (rembg/ISNet)                                 | **Excellent (RMBG-2.0)**                                                    | Good→Excellent                             |
| Quality (photographic elements) | Med    | Adequate (rembg birefnet-lite)                     | **Excellent**                                                               | Good→Excellent                             |
| Apple Silicon reliability       | High   | **Strong (onnxruntime-silicon proven)**            | N/A (cloud)                                                                 | Strong                                     |
| Offline operation               | Low    | **Yes**                                            | No                                                                          | Partial                                    |
| Failure modes                   | Med    | Local model misses → bad alpha                     | Network / quota / cost spike                                                | Both, but each has fallback                |
| **Overall fit**                 | —      | **Strong**                                         | Strong                                                                      | Strongest _long-term_, more code           |

**Key trade-off:** A vs B is "$0 forever and a bit chunkier on edge cases" vs
"~$0.02/call and SOTA edges every time." For a casual skill the user invokes a
few times per week per project, both are fine. C is the long-term answer but
adds branching logic, settings, and a fallback decision tree — more skill to
build, more to maintain.

### Skill ergonomics — user-facing flow

Sketch of the recommended interaction shape (assumes the skill is invoked from a
Claude Code agent turn):

1. **Invocation modes:**
   - `extract-elements <image_path> --markers` (visual markers — auto-detect)
   - `extract-elements <image_path> --describe "mammoth icon upper-left; brown badge bottom-right"`
     (NL mode)
   - `extract-elements <image_path> --points 320,420 800,1100` (click mode, v2)

2. **Phase 1 — Region detection.** Skill detects candidate regions, writes each
   as a numbered preview thumbnail to a temp dir
   (`.tmp/extractions/preview_001.png`, `preview_002.png` …). Each preview shows
   the _cropped region only_ (Shape B's bbox, not yet alpha-cut), with a small
   numeric label burnt into the top-left corner.

3. **Phase 2 — Confirm.** Skill emits a structured response the agent renders in
   chat:

   > Detected 4 candidate regions:
   >
   > 1. ![](preview_001.png) — upper-left, 380×360px → suggested name
   >    `mascot.webp`
   > 2. ![](preview_002.png) — upper-right, 220×220px → suggested name
   >    `favicon.png`
   > 3. ![](preview_003.png) — center, 800×600px → suggested name
   >    `mascot-large.webp`
   > 4. ![](preview_004.png) — bottom-left, 120×40px (looks like a colour swatch
   >    — likely noise?)
   >
   > Accept all? Reject specific numbers? Rename?

   This is exactly the pattern `digestify` already uses in this repo — propose,
   user confirms in one pass — so it's idiomatically consistent.

4. **Phase 3 — Finalize.** For accepted regions, run bg-removal if the user
   chose `--transparent` (default: ask). Write final outputs to `./extractions/`
   (or user-specified `--out`).

5. **Surface:** All preview thumbnails should ALSO be accessible via a tiny
   local HTTP server if the count is high (>6) — reuse the `agent-surface-bun`
   recipe that already powers `tuskboard` and `digestify`. For ≤6 elements,
   inline markdown thumbnails in chat is enough.

**Naming heuristic:** When using NL mode, the user's description per-element
maps 1:1 to filename (slugified). When using markers mode, the agent can do a
follow-up vision call to suggest names ("looks like a mammoth icon →
`mascot.webp`?") — cheap, optional.

**Confirm-before-commit is non-negotiable.** Auto-extraction is too lossy /
inconsistent across models to skip user review. Preview thumbnails + numeric
accept/reject is the lowest-friction way to keep the human in the loop.

### Proof of concept — minimal manual workflow Cole can run today

**Goal:** Validate before building. Spec: ≤30 min, ≤$5 spent, on Cole's actual
Tusk Board branding board.

**Steps:**

1. **(5 min) Pick the target image.** The actual Tusk Board branding board PNG
   that's the motivating use case. Pick three target elements (`mascot.webp`,
   `mascot-large.webp`, `favicon.png`).

2. **(5 min) Validate Claude bbox precision — the gating question.** Open
   claude.ai (or the Anthropic Workbench with Opus 4.7 selected). Upload the
   branding board. Prompt:

   > Return a JSON array of bounding boxes for these elements:
   >
   > 1. The woolly mammoth icon (the main mascot face, head-only or chest-up).
   > 2. The largest woolly mammoth illustration (the body/full pose).
   > 3. The favicon-style square icon (small, square, app-icon-shaped).
   >
   > Format each entry as
   > `{"label": "...", "bbox": [x_min, y_min, x_max, y_max]}` with coordinates
   > in the resized image's coordinate system. Also tell me what resized
   > dimensions you used.

   Then: render those boxes over the original image (any of: open in Preview,
   paste into a Python notebook, or use `sharp` / `vips` to draw them).
   **Eyeball the precision.** Score each box:
   - **Tight** (within ~5% of element edges) → Claude vision alone is enough.
   - **Loose but element-correct** (correctly contains the element, plus 20–50px
     slack) → Good. Use Claude + bg-removal (Shape B).
   - **Wrong element / quadrant-rough only** → Stop. Switch to Florence-2 or
     Grounding DINO via Replicate. Re-run the test there.

3. **(10 min) Validate background removal.** Take the three crops (manually
   cropped at the boxes from step 2; no automation yet). Run them through _one_
   of:
   - **fal.ai Bria RMBG 2.0** (via web UI, no setup) — $0.018 × 3 = $0.054. URL:
     https://fal.ai/models/fal-ai/bria/background/remove
   - **rembg locally** (one-time setup):
     `brew install python@3.11 && uv tool install 'rembg[cli]' && rembg i mascot_crop.png mascot_alpha.png`

   Compare alpha quality side-by-side. Is the matte clean enough that the
   element drops into a Figma/web page convincingly? Pay attention to
   anti-aliased edges and any soft shadows on the source.

4. **(5 min) Validate the marker-detection feasibility shortcut.** Draw coloured
   rectangles in Preview around the three elements on the source image. Save it.
   Use this 8-line OpenCV snippet (Python REPL, `pip install opencv-python`):

   ```python
   import cv2
   img = cv2.imread('marked.png')
   hsv = cv2.cvtColor(img, cv2.COLOR_BGR2HSV)
   mask = cv2.inRange(hsv, (0, 120, 70), (10, 255, 255)) | \
          cv2.inRange(hsv, (170, 120, 70), (180, 255, 255))
   ctrs, _ = cv2.findContours(mask, cv2.RETR_EXTERNAL, cv2.CHAIN_APPROX_SIMPLE)
   boxes = [cv2.boundingRect(c) for c in ctrs if cv2.contourArea(c) > 500]
   for x, y, w, h in boxes:
       cv2.rectangle(img, (x, y), (x+w, y+h), (0, 255, 0), 4)
   cv2.imwrite('detected.png', img)
   ```

   Open `detected.png`. Did the three boxes land cleanly on the three markers?
   If yes — marker mode is a no-brainer to ship.

5. **(5 min) Synthesize.** Three checks pass → confidently build the skill,
   default to Shape B (crop-then-cutout), Claude-bboxes for NL, OpenCV-HSV for
   markers, **rembg local as default bg-remover with fal.ai Bria as an opt-in
   `--quality high` flag.**

**Total cost ceiling:** $0 if using rembg locally, ≤$0.20 if running both
bg-removers through fal.ai. Well under $5.

**Kill criterion:** If step 2 produces bboxes worse than "loose but
element-correct," do not build the NL mode in v1 — ship marker mode only, and
revisit NL when Florence-2 hosted access is integrated.

## Recommendation

- [x] **Create Proposal** — the design space is clear, the risks are bounded,
      and the PoC is cheap enough to gate the build on.

**Rationale:**

1. **The pipeline is well-shaped.** Three independent selection modalities
   feeding two shared post-processing stages (crop → optional alpha) is a clean
   separation that matches three different UX moments without forcing
   architecture sprawl.

2. **The model landscape is mature.** Every stage has at least one local-free
   option and at least one $0.02/call hosted option. Nothing is research- grade.
   Nothing requires CUDA. Nothing requires fine-tuning. Everything has been used
   in production by other people for ≥18 months.

3. **The biggest unknown — Claude bbox precision — has a 15-minute test that
   fully resolves it.** Step 2 of the PoC is the linchpin. Run it before writing
   a single line of skill code. Outcome A (tight boxes) → ship NL mode. Outcome
   B (loose-correct) → ship NL mode with bg-removal absorbing the slack. Outcome
   C (quadrant-rough) → ship markers-only v1, defer NL.

4. **The skill fits the existing pattern.** Marker-detection + bg-removal +
   preview-and-confirm flows naturally into the same agent-surface-bun shell
   that `digestify` and `tuskboard` already use. The `screenshot- optimization`
   skill is a close precedent for "image-in, image-out, run under npx/uv tool,
   no MCP."

5. **Default recommendation for the skill's defaults:**
   - **Region selection:** Markers (OpenCV) for the visual-cue case; Claude
     vision bboxes for the NL case. **No Florence-2/GDINO in v1.**
   - **Mask / cutout:** Shape B (crop-then-cutout) via `rembg` local (default),
     `fal.ai Bria RMBG 2.0` opt-in via `--quality high` flag.
   - **Confirm UX:** Numbered preview thumbnails posted inline in chat for ≤6
     elements; spin up an `agent-surface-bun` mini-page for >6.

**Confidence:** High on architecture and on local-rembg path. Medium on the
NL-via-Claude-bboxes path until step 2 of the PoC runs.
