# Moodboard Element Extraction

**Status:** Draft **Created:** 2026-05-23 **Author:** Cole

---

## Overview

Branding boards / moodboards generated by AI image tools land as a single image
containing many distinct sub-elements — mascot variants, icons, color swatches,
type samples — arranged together on a shared canvas. Extracting individual
assets from these boards today requires manual work in Photoshop or a similar
editor. This proposal scopes an agent skill that automates the extraction step:
user marks the regions they want, the skill cuts them out, optionally removing
background to produce transparent PNGs.

The skill is the first step in a longer arc of "AI generates a board → agent
extracts usable assets" — the Tusk Board project is the immediate beneficiary
(needs `mascot.webp`, `mascot-large.webp`, `favicon.png` from a branding board),
but the workflow generalizes to any future asset generation.

## Problem Statement

Today, after an image-gen agent produces a branding board, getting the
individual elements out is the bottleneck. The user has to open Photoshop or
similar, manually marquee-select each region, copy, paste into a new canvas,
remove background, export. For a 10-element board that's an hour of click work.
The friction is high enough that branding boards often sit unused.

What's needed: a way to tell an agent "extract these elements as separate
images" with minimal manual labor on either side. The agent should do the
mechanical work — pixel-pushing — while the user only needs to specify _what_ to
extract.

## Proposed Solution

A **marker-mode v1**: the user opens their moodboard in any image editor, draws
bright red rectangles around the elements they want extracted, and saves a copy.
The skill detects the rectangles via OpenCV HSV thresholding, crops the interior
of each, and optionally pushes each crop through `rembg` for background removal.
Output: a folder of individual element files, alpha-channel where requested.

This pipeline has been validated end-to-end in this project's `artifacts/`
folder. Cole drew red rectangles on the Tuskboard branding board around three
elements (mammoth icon, coffee-sticker mammoth, magnifier-sticker mammoth). The
detector found all three with pixel-accurate bboxes. `rembg` cleanly removed the
tan canvas background from each, producing transparent PNGs ready for use.

**Why marker-mode and not natural-language selection?** The investigation
([../../investigations/2026-05-23-moodboard-element-extraction-investigation.md])
hypothesized that Claude's own vision could return tight bboxes from prompts
like "the mammoth icon in the top-left". The PoC ran that experiment directly
([poc-findings.md]) and the result was unambiguous: Claude's bboxes on a
multi-element image are **quadrant-rough, not element-rough**. Crops based on
those bboxes routinely contain the wrong element or partial-element. The
investigation's stated kill criterion ("if quadrant-rough, ship markers-only
v1") fired. Marker-mode is what works today; natural-language modes are deferred
to v2+ pending one of the future approaches below.

## Scope

**In Scope (MVP):**

- Marker-mode rectangle detection (OpenCV HSV + contour)
- Crop extraction from clean source or from marked image (with interior inset)
- Optional bg-removal via `rembg` (local, U2Net-based)
- Element naming (numbered by reading-order — user renames as needed)
- Skill packaging consistent with the existing `toolbox` plugin pattern

**Out of Scope (v1):**

- Natural-language selection ("the icon in the upper-left")
- Click-point selection (SAM 2 from point hints)
- Multi-image batch processing
- Hosted bg-removal services (Bria RMBG 2.0 via fal.ai etc.)
- Quality-tier flags (`--quality high` opt-in to a hosted service)
- Interactive preview/confirmation flow inside the skill (post-MVP polish)

**Future Considerations:**

Two paths worth pursuing as follow-up:

1. **Florence-2 / Grounding DINO spike.** The "real" answer to natural-language
   selection. A grounding model designed for this task should produce
   element-tight bboxes from text prompts. Worth a contained spike on a test
   image before committing to a v2 skill mode.

2. **"Nano-banana auto-marker" spike.** An image-editing model (Gemini Flash
   Image / nano-banana) might be able to _draw_ red rectangles directly onto the
   source image when prompted ("draw a red rectangle around each extractable
   element"). This inverts the "report coordinates" failure mode into a
   "manipulate pixels" task that image-editing models are designed for. The
   detector from marker-mode v1 then runs against the model's output unchanged.
   If this works, the user never has to draw anything — the agent auto-marks,
   detects, extracts. High-leverage if it pans out.

3. **Hosted bg-removal upgrade.** Once marker-mode is solid, add a
   `--quality high` flag that routes through fal.ai Bria RMBG 2.0 (~$0.018 per
   crop) for sharper mattes on photographic or fine-edge elements. `rembg` stays
   as the local default for casual use.

## Technical Approach

**Pipeline shape (v1):**

```
marked.png  ──┐
              ├─→ mark_detect.py ──→ list[bbox]
clean.jpg  ──┤                          │
              │                          ▼
              └────────────→ crop_interior(clean, bbox) ──→ crop_NN.png
                                          │
                                          ▼ (optional)
                                    rembg.remove() ──→ crop_NN_alpha.png
```

**Key dependencies:**

- `opencv-python-headless` — HSV thresholding, contour detection
- `rembg` (~176MB model, downloaded on first use) — bg removal
- `Pillow` — image I/O
- Python 3.11+ (the repo's existing skills standardize on Bun, but Python is the
  natural fit for this category given OpenCV + rembg)

**Skill packaging:**

Lives at `plugins/toolbox/skills/moodboard-extractor/`. Two scripts in the
`scripts/` folder mirror the validated PoC: `mark_detect.py` and `bg_remove.py`,
plus an orchestrating `extract.py` that runs both. SKILL.md documents the "draw
red rectangles → run skill" flow with one worked example per common moodboard
type (illustrations, icons, screenshots).

Choice point worth noting: should the skill be **Python** (matching the PoC
tools) or **Bun** (matching the rest of the toolbox)? Python is the right answer
here — OpenCV + rembg are mature Python ecosystems with no comparable Node
equivalents. The `agent-surface-bun` recipe doesn't apply because this is not a
duplex agent/user UI — it's a batch transform.

**Marker detection algorithm (validated):**

1. HSV threshold (red, S≥170, V≥120) — tight enough to ignore the moodboard's
   warm browns and the orange palette swatch.
2. 3×3 morphology close, 1 iteration — bridges pixel-level corner gaps without
   merging adjacent rectangles.
3. `cv2.findContours(RETR_EXTERNAL)` with min bbox area 3000px², aspect ratio
   0.15–8.0.
4. Hollow-rectangle filter: count red pixels inside the bbox; reject if
   `red_pixels / bbox_area > 0.5` (a hollow outline is ~5-15%, a filled red blob
   is ~80-100%).
5. Sort top-to-bottom, then left-to-right, for stable numbering.

## Impact & Risks

**Benefits:**

- Closes the "AI generates → agent extracts" loop, making AI-generated branding
  boards immediately useful
- Marker-mode is deterministic — no surprise outputs, no model spend, no
  hosted-service dependency
- The investigation kill criterion was honored — we're shipping the path the
  evidence supports

**Risks:**

- _Adjacent rectangles can merge into one contour_ — the detector currently
  merges marker rects whose outlines touch or overlap. Mitigation: brief
  SKILL.md note ("leave ~10px gap between adjacent markers"). Real-user
  validation showed Cole's three non-adjacent rects detected cleanly.
- _rembg occasionally produces fringy mattes on photographic crops_ — the v2
  hosted-service upgrade path is the mitigation; for graphics/illustrations the
  local rembg output is clean.
- _Marker color confusion if moodboard contains pure red elements_ — the tight
  HSV filter (S≥170) plus the hollow-rect filter mitigate this, but it's a known
  edge case. Workaround: switch marker color (skill flag
  `--marker green|blue|red`).

**Complexity:** Low — the PoC already validates every pipeline stage. The
remaining work is packaging into a proper skill with documentation, error
handling, and tests.

## Open Questions

- **Should v1 require both marked + clean inputs, or accept just the marked
  image?** Both-inputs gives cleanest output (no risk of cropping red marker
  pixels). Just-marked is one-step easier for the user (inset cropping pushes
  inside the marker lines). Currently the script supports both with `--clean`
  optional. Lean: just-marked is the default with a note about inset trade-off;
  users who care about pixel-perfect edges supply `--clean`.
- **Skill naming.** "moodboard-extractor" is descriptive; "asset-extractor" is
  more general but vague. Pick one for skill folder + frontmatter.
- **Should we ship the nano-banana spike inside this same project, or split it
  into a follow-up project once v1 is delivered?** Lean: ship v1 first, open a
  follow-up project for the auto-marker spike.

## Success Criteria

- Skill packaged at `plugins/toolbox/skills/moodboard-extractor/` and follows
  the repo's existing skill conventions (SKILL.md, scripts/, optional dist/,
  tests).
- Cole can extract `mascot.webp`, `mascot-large.webp`, `favicon.png` from a
  Tuskboard-style branding board in <2 minutes of his time (draw rectangles,
  invoke skill, rename outputs).
- bg-removal produces visually-acceptable transparency for at least the
  illustration-style elements (stickers, mascot, icons).
