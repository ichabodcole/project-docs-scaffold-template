# UX Review: Tusk Board (tuskboard) skill pre-ship assessment

## Metadata

- **Date:** 2026-05-22
- **Report Type:** Skill UX / Documentation Review
- **Status:** Completed
- **Generated By:** Claude Opus 4.7 (acting as a first-time agent user of the
  skill, in a live test session)
- **Scope:** `plugins/toolbox/skills/tuskboard/SKILL.md` and supporting scripts
  (`bg.ts`, `server.ts`, `join.ts`), exercised end-to-end in **monitored** host
  mode

---

## Executive Summary

The Tusk Board skill is in good shape and the duplex agent ↔ user loop
genuinely works as advertised. Setup was quick, the monitored host mode felt
close to real-time, and the protocol surface is small enough to internalize in
one reading.

The frictions surfaced are all documentation-level — no code changes are
implied. The skill is ship-ready; these are polish notes for the next pass.

**Key Findings:**

- Monitored host mode (`bg.ts` + `Monitor`) is ergonomic and reliable in
  practice — the file-mediated transport felt invisible.
- The verbatim "worked example tool-call sequence" in the SKILL is the single
  most load-bearing section; setup was near-zero-friction because of it.
- Three small but real frictions appeared on first use: a race when reading the
  meta line, an unset `$CLAUDE_PLUGIN_ROOT` env var, and the long inline
  `Monitor` grep regex.

**Top Recommendations:**

1. Document the `until [ -s file ]` wait pattern for reading the meta line.
2. Add a one-liner about substituting an absolute path when
   `$CLAUDE_PLUGIN_ROOT` is unset.
3. Show a worked `task.update` example responding to a user-initiated `task.add`
   — common pattern, currently inferred.

**Overall Assessment:** Ship it. The remaining items are 5-minute doc tweaks,
not blockers.

## Findings

### High-value strengths (keep these)

#### Finding: The duplex loop feels real-time in practice

**Description:** Each user board action (drag, add, edit, toggle) arrived as a
clean `task-notification` carrying the event line, and replying via
`appendFileSync` on the cmds file was a one-liner each time.

**Impact:** This is the core promise of the skill, and it lands. No glue code,
no state to carry between turns beyond the two file paths.

**Evidence:** Multiple round-trips during the test — drag-to-done, user-added
tasks, toast acknowledgements — all fired with no perceptible lag and no lost
events.

**Recommendation:** Preserve this exact contract. The agent-appends-cmds-file /
agent-reads-events-via-Monitor split is the right shape.

---

#### Finding: The "Worked example — verbatim tool-call sequence" section is the doc's MVP

**Description:** The Turn 1 / Turn 2..N / Final turn block in `SKILL.md` lines
313-370 made setup near-frictionless. Without it, the Monitor grep regex (with
its escaping) would have been a likely typo source.

**Impact:** First-use success rate hinges on this section. It is what made me
get the Monitor armed correctly on the first try.

**Recommendation:** Treat this section as load-bearing. Any future edits to the
protocol should update this example **first**, then the prose.

---

#### Finding: `message` (toast) is delightful and underadvertised

**Description:** The `message` command for toasts adds noticeable personality to
the duplex interaction and was the easiest way to acknowledge user moves without
polluting the task list.

**Impact:** UX-wise this is one of the highest-value primitives; it is buried in
a protocol table.

**Recommendation:** Promote it in the host-mode intro with a one-line callout:
"Use `message` for lightweight acknowledgements — toasts on the board, no task
list churn." See [Recommendation R7](#medium-priority).

---

### Medium-friction issues (doc-level)

#### Finding: Meta-line read is racy without an explicit wait pattern

**Description:** The SKILL says "Wait briefly, then read /tmp/bg-meta.json —
first line is the meta JSON" (lines 325-326). With `run_in_background: true`,
the meta line is not guaranteed to be written by the time the agent reads the
file. A blind `cat` will return empty content.

**Impact:** First-run agents will either fail silently (empty meta) or insert
arbitrary `sleep` calls. Both are bad outcomes.

**Evidence:** During this test I used
`until [ -s /tmp/tusk-bg-meta.log ]; do sleep 0.2; done; head -1 …` and it
worked first try.

**Recommendation:** Replace "wait briefly" with the `until [ -s file ]` pattern
in the worked example. One-line fix.

---

#### Finding: `$CLAUDE_PLUGIN_ROOT` is not always set in the agent's Bash env

**Description:** All invocation lines use
`bun run ${CLAUDE_PLUGIN_ROOT}/skills/tuskboard/scripts/bg.ts`. In this session,
`$CLAUDE_PLUGIN_ROOT` resolved to the empty string, which silently turns the
path into an invalid `/skills/...` and the `bun run` fails with an unhelpful
"module not found".

**Impact:** A new agent may not realize the variable is unset; the failure
message points at the path, not the env var.

**Evidence:** `echo "CLAUDE_PLUGIN_ROOT=$CLAUDE_PLUGIN_ROOT"` → empty in my Bash
tool environment.

**Recommendation:** Add a one-line note near the first invocation example: "If
`$CLAUDE_PLUGIN_ROOT` is not set in your shell, substitute the absolute path to
the plugin (e.g. `plugins/toolbox/skills/tuskboard/scripts/bg.ts` relative to
the repo root)."

---

#### Finding: Inline `Monitor` grep regex is long and easy to typo

**Description:** The grep filter
`'"type":"task\\.(add|move|toggle|edit|remove)"|"type":"submit"|"type":"closed"'`
inlined into a JSON tool call needs careful escaping. It is correct in the doc,
but mistakes are easy in transcription.

**Impact:** A wrong filter either misses events or produces an over-broad stream
that the harness will rate-limit.

**Recommendation:** Ship a small helper next to the other scripts, e.g.
`scripts/watch-events.sh <events_file>`, that runs the canonical
`tail -F | grep` pipeline. Then the Monitor command becomes a literal
`bash scripts/watch-events.sh /tmp/…log`. Reduces escape pain and lets the
filter evolve without doc edits.

---

#### Finding: No worked example for the "agent reacts to a user-added task" pattern

**Description:** The host protocol shows `task.add` events from the user and a
`task.update` command from the agent, but does not show the natural pairing
("user adds a task → I move it / annotate it"). I had to infer that the agent
reuses the id from the user-emitted `task.add` event.

**Impact:** Common-case pattern; easily missed by a first-time agent that only
reads the protocol tables.

**Recommendation:** Add a 3-line snippet under the worked example: a
notification with `task.add` and the matching `task.update` reply that
references the same id.

---

#### Finding: Orphan-timeout behavior deserves an earlier mention

**Description:** The "Failure modes" note under `bg.ts` documents that closing
the browser tab leaves an orphan host until the 30-minute idle timeout. This is
the correct behavior, but it is buried in the Notes block (lines 415-420).

**Impact:** Users will reasonably assume "closed tab = session ended". Without
an earlier heads-up, agents may report a stale state on resumed sessions.

**Recommendation:** Add a one-line callout in the **Static vs Monitored** pick
section, e.g. "Monitored sessions survive tab close — the host stays alive until
the user submits, cancels, or the idle timeout (default 30m) fires."

---

### Low-priority observations

#### Finding: Discovery file `tuskboard-latest.json` could collide silently

**Description:** Two `bg.ts` instances spawned in close succession will both
overwrite `tuskboard-latest.json`. The doc calls this out (line 411-414), but
provides no remediation other than "prefer explicit `--id`".

**Impact:** Low — multi-board host setups are rare and the workaround is
documented. Just worth a friendly default.

**Recommendation:** Optional follow-up: have `bg.ts` refuse to overwrite a
`latest.json` newer than N seconds without `--force`, or append a session
suffix.

---

#### Finding: The exit-code table is clear but does not name "closed by close-command"

**Description:** The table covers submit / bad input / timeout / user-cancel but
does not explicitly map the case where the agent appended `{"type":"close"}` to
the cmds file. The prose elsewhere says "exit 0" — make the table match.

**Impact:** Cosmetic only.

**Recommendation:** Add a row or merge into the "Submitted (or agent sent
close)" row already present, which actually does cover it — could be made more
explicit.

## Recommendations

### High Priority

1. **R1 — Replace "wait briefly" with explicit `until [ -s file ]` wait in the
   worked example**
   - Action: Edit the Turn 1 block in `SKILL.md` to show the wait loop.
   - Rationale: First-run setup correctness; closes a silent-failure path.
   - Estimated Effort: Quick win (<5 min).
   - Follow-up: Direct edit.

1. **R2 — Add a `$CLAUDE_PLUGIN_ROOT` fallback note near the first invocation**
   - Action: One-line callout: "if unset, substitute the absolute path".
   - Rationale: Empty-var silent failure is unhelpful; cheap to prevent.
   - Estimated Effort: Quick win.
   - Follow-up: Direct edit.

### Medium Priority

1. **R3 — Show a worked `user task.add → agent task.update` pattern**
   - Action: Add a 3-line snippet under the worked example.
   - Rationale: Most common reactive pattern; should not require inference.
   - Estimated Effort: Quick win.

1. **R4 — Ship a `watch-events.sh` helper to replace the inline Monitor grep**
   - Action: Add `plugins/toolbox/skills/tuskboard/scripts/watch-events.sh` and
     reference it from the Monitor invocation example.
   - Rationale: Reduces escape pain, lets the filter evolve without doc edits.
   - Estimated Effort: ~30 min (script + doc update + smoke test).
   - Follow-up: Direct action.

1. **R5 — Move the orphan-timeout heads-up earlier in the doc**
   - Action: One-line callout in the Static vs Monitored picker section.
   - Rationale: Surfaces a behavior new users will misjudge by default.
   - Estimated Effort: Quick win.

1. **R6 — Tighten the exit-code table to explicitly cover agent-issued `close`**
   - Action: Update the "Submitted (or agent sent close)" row wording.
   - Rationale: Removes ambiguity.
   - Estimated Effort: Quick win.

### Low Priority / Monitor

1. **R7 — Promote `message` (toast) more prominently in the host-mode intro**
   - Action: One-line callout that highlights it before the protocol table.
   - Rationale: It is one of the highest-leverage UX primitives in the skill and
     easy to overlook.

1. **R8 — Consider a friendlier `tuskboard-latest.json` collision behavior**
   - Action: Optional — add a recency guard or document the explicit-`--id`
     guidance more prominently.
   - Rationale: Rare collision, but cheap to make defensive.

## Methodology

**Tools Used:**

- The tuskboard skill itself (`bg.ts`, `Monitor`, `cmds_file` / `events_file`),
  exercised live.
- `bun -e` one-liners to drive the cmds file.

**Process:**

1. Read `SKILL.md` end-to-end.
2. Spawned `bg.ts` in the background and read the meta line.
3. Armed a Monitor on the events file with the documented grep filter.
4. Seeded six realistic tasks via an `init` command.
5. Reacted in turn-by-turn loops to the user's drags, adds, and edits.
6. Captured friction points in real time and posted them back to the board.
7. Wrote up the resulting findings here.

**Limitations:**

- Single session, single agent acting as "user".
- Static and join modes were not exercised in this test.
- No code-level review of the underlying scripts.
